# MCP Mode: Inverted-Control Agent Execution

**Date:** 2026-05-28
**Status:** Design — L1 config complete; L2 tool surface and L3 execution pending

## Problem

The current execution model is headless: when a prompt step fires, `cloched`
launches `claude -p` inside the container, waits for it to exit, and reads
`CLOCHE_RESULT:<name>` from stdout.  This works for fully automated pipelines
but breaks interactive use — a developer cannot connect an *interactive* Claude
session and have the workflow wait for their completion.

MCP mode inverts the control: instead of the daemon launching an agent, an
interactive Claude client connects to the daemon's MCP server, pulls pending
prompt steps one at a time, executes them interactively, and pushes results
back.  Script steps are unaffected.

---

## Decision: where the MCP server runs

**The MCP server runs in the existing daemon (`cloched`) at the gRPC/HTTP
listen address.**

Alternatives considered:

| Option | Notes | Rejected because |
|---|---|---|
| MCP server in the agent container (`cloche-agent`) | Agent containers have no stable network address; they're ephemeral and behind Docker NAT | Not reachable from developer laptop |
| Separate `cloche-mcp` sidecar process | Requires separate lifecycle management and adds a new network hop to the run-state store | Operational burden with no benefit |
| MCP server in daemon (chosen) | Daemon already holds run state, step queue, and KV store; reachable at `CLOCHE_ADDR` | — |

**Reachability:** Claude (running on the developer's laptop or inside a
container) connects to the daemon at `CLOCHE_ADDR` (default
`localhost:50051`, or `host.docker.internal:50051` from inside a container).
This is the same address already used by `cloche-agent` and the `clo` CLI.
No firewall changes are required for local development.

**Transport:** The MCP server is exposed over HTTP (Server-Sent Events or
streamable-HTTP, as required by the MCP spec) alongside the existing
`cloched` HTTP dashboard port.  A dedicated `/mcp` path prefix keeps it
separate from the web UI.  The gRPC port is unaffected.

## MCP tool surface

Three tools are exposed.  All require a valid `session_token` except `init`.

### `init`

Register the connecting client as the interactive agent for a specific run.

```jsonc
// Request
{
  "run_id": "string"   // the cloche run ID (e.g. "run-abc123")
}

// Response
{
  "session_token": "string"   // opaque token; present this on subsequent calls
}
```

- Returns a fresh token on each call.  A second `init` for the same run
  replaces the existing session.
- The daemon starts a connect-timeout countdown when the run reaches its first
  prompt step.  `init` must arrive before this timeout expires (see
  Configuration).

### `next`

Poll for the next pending prompt step.  Blocks until a step is available or
the run ends.

```jsonc
// Request
{
  "session_token": "string"
}

// Response — step available
{
  "step_name": "string",
  "prompt":    "string"   // fully rendered prompt text
}

// Response — run complete (no more steps)
{
  "done": true
}
```

- If no step is currently waiting, `next` long-polls (up to 30 s) before
  returning `{"done": false}` so the client can retry.  This avoids busy-wait
  but keeps the MCP connection alive.
- Only prompt steps (`type = agent` without a `run:` config key) appear here.
  Script steps run headlessly in the container as before.

### `submit-result`

Signal that the interactive agent has finished a prompt step and provide the
result and token usage.

```jsonc
// Request
{
  "session_token":  "string",
  "step_name":      "string",
  "result":         "string",   // named result, e.g. "success", "fail", "abort"
  "output":         "string",   // human-readable step output / summary (optional)
  "output_tokens":  42          // integer; used for cost accounting
}

// Response
{
  "ok": true
}
```

- `result` must be one of the step's declared result names, or a built-in
  terminal (`done`, `abort`).  The daemon rejects unknown result names with an
  error.
- After a valid `submit-result`, the daemon unblocks the engine's `Execute`
  call for that step and records the result identically to a gRPC `StepResult`
  message.
- `output` is written to the step's `.log` file in the run workspace.
- `output_tokens` is recorded to the `step_executions` table alongside the
  step result (same schema as prompt-mode token usage).

## Control flow sequence

```
Developer laptop                daemon (cloched)            cloche-agent container
──────────────────              ────────────────            ──────────────────────
cloche run <workflow>
                          →     create run, start container
                                container connects (AgentSession gRPC)
                                engine hits script step
                                                    →       execute script, send StepResult
                          ←     record script result
                                engine hits prompt step
                                park step: mcpSessionStore.Wait(runID, stepName)
Claude connects (MCP)
 → init(run_id)           →     validate run_id, mint session_token
 ← {session_token}        ←
 → next(session_token)    →     dequeue parked step
 ← {step_name, prompt}    ←
 [Claude works...]
 → submit-result(...)     →     validate result name
                                unblock mcpSessionStore.Wait → StepResult
                                record result + tokens in DB
                                engine advances to next step
                                if next step = script → dispatch to agent
                                if next step = prompt → park, wait for next()
 → next(session_token)    →     dequeue next parked step (or done)
 ← {step_name, prompt}
  OR {done: true}         ←
```

The `cloche-agent` container stays connected for the duration of the run and
handles all script steps normally.  It never sees prompt steps in MCP mode —
those are consumed by the MCP client.

## Result protocol

In prompt mode, `claude -p` writes `CLOCHE_RESULT:<name>` to stdout;
`internal/protocol.ExtractResult` parses it.

In MCP mode, this stdout marker is **replaced** by the `result` parameter of
`submit-result`.  The daemon records it via the same `domain.StepResult` path
used for gRPC results — no change to the persistence layer is needed.

Named results from `submit-result` are validated against the step's declared
`results` list (in the workflow DSL).  The built-in terminals (`done`, `abort`)
are always accepted.  The daemon returns an error response for unrecognised
names.

Token usage flows from `submit-result.output_tokens` directly into the
`TokenUsage` struct (`OutputTokens` field); `InputTokens` is left zero because
the MCP client is responsible for its own prompt construction and the daemon
has no visibility into Claude's context window.

## Integration with AgentAdapter.Execute

`internal/ports/agent.go` defines:

```go
type AgentAdapter interface {
    Name() string
    Execute(ctx context.Context, step *domain.Step, workDir string) (domain.StepResult, error)
}
```

A new `internal/adapters/agents/mcp.Adapter` will implement this interface for
prompt steps in MCP mode.  Its `Execute` method:

1. Registers the step in the daemon's `mcpSessionStore` (an in-process map of
   `runID+stepName → chan domain.StepResult`).
2. Blocks on the channel until a `submit-result` call fires it.
3. Returns the `domain.StepResult` (result name + usage).
4. If the context is cancelled before `submit-result` arrives, returns an error
   so the engine can fail the step.

The adapter is selected at dispatch time: the engine checks
`config.Agent.Mode == "mcp"` and routes prompt steps to `mcp.Adapter` instead
of `prompt.Adapter`.  Script steps always go to `generic.Adapter` regardless
of mode.

## Configuration

```toml
# .cloche/config.toml  (per-project)
[agent]
mode = "mcp"              # "prompt" (default) or "mcp"
connect_timeout = "120s"  # how long to wait for init() before failing run
                          # (default: 120s; L2 implementation detail)
```

`mode` is the only field required for L1.  `connect_timeout` is deferred to
the L2 implementation.

## What's deferred

| Item | Deferred to |
|---|---|
| `mcpSessionStore` implementation (in-memory, thread-safe map + channels) | L2 |
| HTTP MCP endpoint wiring (`/mcp` on the HTTP port) | L2 |
| Tool schema registration (MCP SDK integration) | L2 |
| `mcp.Adapter` implementing `AgentAdapter` | L2 |
| Connect-timeout enforcement and failure path | L2 |
| Engine mode-selection logic (route to mcp.Adapter vs prompt.Adapter) | L3 |
| End-to-end run execution with a real MCP client | L3 |
| Conversation continuity across prompt steps | future |
| Multi-client concurrency (one client per run is the initial constraint) | future |
| Token log flow (streaming output tokens back through MCP) | future |

## Open questions resolved

**Q: Does the MCP client need to know the run ID before connecting?**
Yes.  The `init` tool takes `run_id` explicitly.  The developer obtains it
from `cloche run --print-id` or from the dashboard URL.  This avoids ambiguity
when multiple runs are active.

**Q: What happens if `submit-result` is called for a step that isn't waiting?**
The daemon returns an error: `{"error": "no pending step <name> for run <id>"}`.
The client should retry `next` to re-sync state.

**Q: How does session continuity work across prompt steps?**
The same `session_token` is reused across all `next`/`submit-result` calls
within a run.  The MCP transport layer (SSE or streamable-HTTP) stays open.
Conversation continuity within Claude's context window is the client's
responsibility and is not managed by the daemon.
